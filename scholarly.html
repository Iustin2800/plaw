Introduction

In this document we will present a multi-user education game for learning in a fun way important concepts regarding Web technologies.

This has a purely academic purpose. This document:

  • contains a list of relevant sketches, wireframes, storyboards and interactive UI prototypes designed for Web and Mobile
  • discusses the usability of interaction types/paradigms
  • provides a number of personas that suits our application
  • considers how interaction design patterns and used in the presentation of wireframes/prototypes.
  • discusses the user flows and affective factors and emotions.

Old fashioned ways of teach are no longer as effective as they used to be, new generations are more willingly to have fun rather than study and they lose interest much faster than generations before. In order to keep them focused and in schools, teachers had to meet them half-way and change their teaching style to somewhat more fun. As every problem has at least a solution, people came up with applications that are using games a way to deliver information and teach.

Name and logo

"Plaw" represents the abbreviation of the application's name "Playing the Web". The logo achieves a distinct and memorable identity that sets it apart from other names. The use of abbrevations also facillitates effortless communication among users, making it users for them to refer to the app in the conversations with one another.

Talking about the logo itself, it is a globe with a graduate hat, represents the main reason of our application, which is to facilitate educational activities. This symbol also signifies application's focus on teaching web techonlogies beside the commitment to providing a platform for learning.

Color scheme

The color scheme mainly formed of the main turqoise color, is easy to the eye and very welcoming, it's also kind of a fun color, representing the fun aspect of the learning through games; there are also other complementing colors, which are bright and diverge focus on something important/needed.

Typography

The font (Hind) is a simple and basic one, it's not an out of the ordinary font as to not sacrifice readability; after all, the app's purpose is to teach users through games, and text is very important in the learning part

Affective factors and emotions

We want to promote our application as a way of learning while having fun, so you won't feel like putting much effort into acquiring that information. On the other way, we want our users to feel challenged mentally and in order to do so we added a level of competition by letting you race against other players.

Since we know how learning process can be and we do no want our users to lose their interest, we developped a "Featured" menu where you can see the popular games, new games and most liked games.

Interaction design patterns

Web

The homepage of the application has a simple navigation with no menus. The layout grid is card type, within there are information regarding the content of the website.

Once logged into our account we can see that the content is following a split type layout grid: in the left part we have games, information about your account and rankings; in the right part there is the social part where you can find all your friends and groups. Once again the left layout is split in two parts, each part being aligned.(left to the left and right to the right) All buttons are hiding behind actions that can be triggered by left clicking. Also every person/group you see in the social part can be interacted with by left clicking which will open a chat.

As you can see, we are using card layout to keep information short and concrete, so our users would not lose interest while browsing our page. In the "games" card and "ranking" card we used an expandable list which expands the list of the games/players you see until you've reached the end of that list

Selecting a game will open up a new page with a similar layout as before. On the left we have a card with informations useful while playing the game and on the right we have the game in the top and a console in the bottom. Here we have a form pattern for input text.

Mobile

The bottom bar represents the navigation menu and is formed of icon buttons representing different tabs, there is also a floating setting button at the top right of the app, always there no matter when someone switches a tab or scrolls up and down.

The layout is kept simple, everything is organized in a scrollable list with many items of different sizes placed within, from which the user can interact with.

There also is an intuitive tab switching feature using simple left and right swipes, there are also buttons to access those tabs for people who can't or don't want to swipe, everything else consists of buttons that require simple taps for simplicity.

Interaction types/paradigms

One of them would be traditional interaction which seems to be the most common interaction type among our users. Our application being on a website it would be easier for anyone to use a mouse and keyboard in order to use our aplication because we have few buttons to interact with and many activities that include text input from users.

However, it doesn't mean there is no space for others interaction types like haptic interactions. As we are also developping our application on mobile, we can use vibrations as feedback for wrong or incomplete answers to the courses. We can also have a resistance sensation to indicate that a puzzle piece won't fit in the place our user intended to place it, or it can be used for the end of the game to indicate that the game is over and any other interactions will not be effective.

As we are talking about mobiles, there is also room for gestural interaction such as swiping through menus or courses which can help navigation by a lot. Combing gestural and haptic interactions in order to give the best feedback to our mobile users. For example: a tutorial will let user interact with a certain object and he would be asked to move it in order to fit somewhere. The user will have a little display with the code of that object.If our user doesn't manage to do the task, he would get slight vibrations to indicate that is wrong and for every wrong move the object will be slightly resistance to simulate the sensation of touching. If he does manage to do it, he will see the code modified.

When it comes to the next type of interaction it doesn't matter if it is web or mobile application because social interaction is cross-platform. You can find new friends, have a little chat and also create groups where you can add your entire social group.
